1. PROJECT SUMMARY
================================================================================
A custom wedding website built for Ashlyn & Jeffrey Paine's wedding on
September 26, 2026 at Davis & Grey Farms in Celeste, TX. The site is a full-
stack Next.js application with a Supabase PostgreSQL backend, deployed on Vercel.

Core features:
- Public wedding website (10 pages covering all wedding info)
- RSVP system (household-based lookup, meal options, extras)
- Admin dashboard (guest management, RSVP analytics, content editing)
- Admin edit mode (in-page image/text editing via floating toolbar)
- Three wedding games: Painedle (daily Wordle), Couple Trivia, Mini Crossword
- Games leaderboard system with guest accounts stored in localStorage

The site is production-ready with real content partially populated. Guest list
(~192 guests, 103 households) is seeded via SQL migration.

Repository location: /Users/jeffpaine/Documents/Antigravity/ThePaineWedding/
Dev worktree (AI sessions): /Users/jeffpaine/.claude/worktrees/sweet-robinson/

  The Supabase service role key (NEVER expose to client)
  Used by: all server-side API routes that write or read admin data
  Files: site-settings route, upload-image route, admin/trivia-questions routes,
         admin/auth route (admin_logs insert), admin/session route,
         site-settings.ts (getWeddingData), submit-score route

ADMIN_SESSION_SECRET
  A strong random string used to HMAC-sign session cookies
  If missing, falls back to concatenation of all ADMIN_PASSWORD_* values
  Recommended: 32+ random hex characters

ADMIN_PASSWORD_MASTER
  Master admin password. Grants full access including:
  - Security logs tab
  - AdminEditBar edit mode
  - All admin features

ADMIN_PASSWORD_1 through ADMIN_PASSWORD_5
  Standard admin passwords. Grant access to:
  - RSVP dashboard
  - Games admin
  - Content admin
  But NOT: security logs, AdminEditBar edit mode

Note: The session secret falls back to concatenating all passwords if
ADMIN_SESSION_SECRET is missing. Always set ADMIN_SESSION_SECRET explicitly.

--- Migration 0: 0000_schema.sql ---
Base schema created at project start.

TABLE: households
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid()
  name        TEXT NOT NULL UNIQUE  (e.g. "The Paine Family 1", "The Lawson Family")
  created_at  TIMESTAMPTZ DEFAULT now()

TABLE: guests
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid()
  household_id    UUID REFERENCES households(id) ON DELETE CASCADE
  first_name      TEXT NOT NULL
  last_name       TEXT NOT NULL
  suffix          TEXT         (nullable, e.g. "Jr.", "III")
  nicknames       TEXT[]       (array of alternate names for search)
  attending       BOOLEAN      (null = not responded, true = yes, false = declined)
  meal_option     TEXT         (nullable, meal choice)
  created_at      TIMESTAMPTZ DEFAULT now()
  updated_at      TIMESTAMPTZ DEFAULT now()

TABLE: admin_logs
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid()
  event       TEXT NOT NULL    (e.g. "login_success", "login_failure")
  role        TEXT             (nullable, admin role if successful)
  ip          TEXT             (nullable)
  user_agent  TEXT             (nullable)
  created_at  TIMESTAMPTZ DEFAULT now()

RLS: Currently DISABLED on households and guests. This is a known gap.
All RSVP reads/writes go through client-side Supabase with anon key.

--- Migration 2: 20260308010000_add_game_leaderboards.sql ---
TABLE: game_players
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid()
  email       TEXT NOT NULL UNIQUE
  username    TEXT NOT NULL
  created_at  TIMESTAMPTZ DEFAULT now()
  updated_at  TIMESTAMPTZ DEFAULT now()

TABLE: game_scores
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid()

--- CRITICAL TAILWIND V4 QUIRK ---
Tailwind v4 uses CSS-first configuration. The @theme {} block defines custom
tokens. However, font utilities CANNOT use CSS var() references inside @theme
at computed-value time — this causes IACVT (Invalid at Computed Value Time).

WRONG (breaks in v4):
  @theme {
    --font-heading: var(--font-playfair), serif;  /* BREAKS */
  }

CORRECT (working solution):
  Define font utilities in @layer utilities instead:
  @layer utilities {
    .font-heading { font-family: var(--font-playfair), serif; }
    .font-body { font-family: var(--font-inter), sans-serif; }
  }

This is defined in src/app/globals.css.

9. SITE SETTINGS SYSTEM — DB OVERRIDES
================================================================================
File: src/lib/site-settings.ts (server-only — uses service role key)

PURPOSE: Allow the admin to change images, text, and content without redeploying.
The site_settings table stores key/value pairs that OVERRIDE static defaults from
wedding-data.ts. The getWeddingData() function merges DB values over static ones.

getWeddingData() returns:
  {
    wedding:     (merged wedding data object),
    images:      (merged images object),
    content:     { homeIntro, storySubtitle },
    overlays:    { hero: ImageOverlay|null, attireLadies: array, attireGents: array },
    settingsMap: (raw key/value map from DB for admin panel use)
  }

All server components that need dynamic content call getWeddingData().
This function uses the SERVICE ROLE KEY and must only run server-side.

After any site_settings change, the API route calls:
  revalidatePath("/", "layout")
This immediately invalidates all cached pages and forces fresh renders.

--- Supported site_settings Keys ---

IMAGES:
  "images.hero"              → { main: string, overlay: { color, opacity }|null }

--- POST /api/admin/auth ---
File: src/app/api/admin/auth/route.ts
PURPOSE: Admin password login. Validates password, creates session cookie.
AUTH: None required (this IS the auth endpoint)
INPUT (JSON body): { password: string }
PROCESS:
  1. Builds PASSWORD_MAP from env vars:
       ADMIN_PASSWORD_MASTER → role "Master"
       ADMIN_PASSWORD_1      → role "User 1"
       ADMIN_PASSWORD_2      → role "User 2"
       ...through ADMIN_PASSWORD_5
  2. Constant-time comparison to prevent timing attacks
  3. On failure: waits fixed delay (prevents brute-force timing)
  4. Logs attempt to admin_logs table (event: login_success or login_failure)
  5. On success: creates HMAC-signed session token, sets cookie:
       Name:     wedding_admin_session
       HttpOnly: true
       Secure:   true (in production)
       SameSite: lax
       MaxAge:   14 days
OUTPUT: { success: boolean, role?: string, error?: string }

--- GET /api/admin/session ---
File: src/app/api/admin/session/route.ts
PURPOSE: Verify current session and return role.
AUTH: Reads wedding_admin_session cookie
OUTPUT: { role: "Master"|"User 1"...|"User 5" } or 401

--- DELETE /api/admin/session ---
Same file.
PURPOSE: Logout. Clears the session cookie.
OUTPUT: { success: true }

--- GET /api/admin/site-settings ---
File: src/app/api/admin/site-settings/route.ts
PURPOSE: Fetch all current site settings for admin panel display.
AUTH: Requires valid admin session cookie
OUTPUT: { settings: Record<string, unknown> }

--- POST /api/admin/site-settings ---
Same file.
PURPOSE: Upsert one setting (create or update).
AUTH: Requires valid admin session cookie
INPUT (JSON body): { key: string, value: unknown }
PROCESS: Upserts row in site_settings; calls revalidatePath("/", "layout")
OUTPUT: { success: true }

--- DELETE /api/admin/site-settings ---
Same file.
PURPOSE: Remove one setting (restore to static default).
AUTH: Requires valid admin session cookie
INPUT (JSON body): { key: string }
PROCESS: Deletes row in site_settings; calls revalidatePath("/", "layout")
OUTPUT: { success: true }

--- POST /api/admin/upload-image ---
File: src/app/api/admin/upload-image/route.ts
PURPOSE: Upload an image to Supabase Storage.
AUTH: Requires valid admin session cookie
INPUT: multipart/form-data with:
  file:  the image file (max 10MB per bucket policy)
  path:  (optional) storage path prefix, e.g. "story/item-0"
PROCESS:
  1. Reads file from form data
  2. Generates unique filename using Date.now() + original extension
  3. Uploads to site-images bucket in Supabase Storage
  4. Returns public URL
NOTE: Client pre-compresses images to max 2400px, 0.92 JPEG quality before
sending to stay under Vercel's 4.5MB request body limit.
OUTPUT: { url: string, path: string }

--- GET /api/admin/trivia-questions ---
File: src/app/api/admin/trivia-questions/route.ts
PURPOSE: Fetch ALL trivia questions including archived (admin view).
AUTH: Requires valid admin session cookie
PROCESS: SELECT * FROM trivia_questions ORDER BY sort_order ASC (service role)
OUTPUT: { questions: TriviaQuestionRow[] }

--- POST /api/admin/trivia-questions ---
Same file.
PURPOSE: Create a new trivia question.
AUTH: Requires valid admin session cookie
INPUT (JSON body): { prompt, answer_a, answer_b, answer_c, answer_d,
                     correct_index (0-3), fun_fact?, sort_order? }
VALIDATION: All 4 answers + prompt required; correct_index must be 0–3
OUTPUT: { question: TriviaQuestionRow }

--- PUT /api/admin/trivia-questions/[id] ---
File: src/app/api/admin/trivia-questions/[id]/route.ts
PURPOSE: Update an existing trivia question (partial update OK).
AUTH: Requires valid admin session cookie
INPUT (JSON body): any subset of { prompt, answer_a, answer_b, answer_c,
                   answer_d, correct_index, fun_fact, sort_order, archived }
OUTPUT: { question: TriviaQuestionRow }

--- DELETE /api/admin/trivia-questions/[id] ---
Same file.
PURPOSE: Hard-delete a trivia question.
AUTH: Requires valid admin session cookie
OUTPUT: { success: true }

--- POST /api/games/submit-score ---
File: src/app/api/games/submit-score/route.ts
PURPOSE: Submit or update a player's game score.
AUTH: None (public, but rate-limiting via score comparison)
INPUT (JSON body):
  { game, score, maxScore, attempts?, solved, puzzleKey, email, username,
    metadata?: { ... browser profile fields ... } }
PROCESS:
  1. Upserts game_player record by email (creates or updates username)
  2. Looks up existing score for (player_id, game, puzzle_key)
  3. Only updates if new score is better:
       isBetterScore() = new score > old score
                      OR (same score AND fewer attempts)
  4. Captures request metadata: X-Forwarded-For (IP), CF-IPCountry (country),
     CF-IPRegion (region), CF-IPCity (city), User-Agent, Accept-Language,
     Referer headers — stored in score's metadata JSONB column
OUTPUT: { success: true, improved: boolean }

--- POST /api/games/validate-word ---
File: src/app/api/games/validate-word/route.ts
PURPOSE: Check if a Painedle guess is a valid English word.
AUTH: None (public)
INPUT (JSON body): { word: string }
PROCESS: Calls isValidDictionaryGuess(word) from lib/games/dictionary.ts
  Dictionary loads the npm "word-list" package (500k+ English words) into a
  cached Set on first call. Also accepts all 80 Painedle puzzle words.
OUTPUT: { valid: boolean }

--- GET /api/games/trivia-questions ---
File: src/app/api/games/trivia-questions/route.ts
PURPOSE: Public fetch of active trivia questions for the game UI.
AUTH: None (public)
CACHE: export const revalidate = 60 (ISR, 60-second cache)
PROCESS: SELECT * FROM trivia_questions WHERE archived = false
         ORDER BY sort_order ASC (uses anon key)
OUTPUT: { questions: TriviaQuestionRow[] }

15. RSVP SYSTEM — FULL DETAIL
================================================================================

--- HOUSEHOLD MODEL ---
Every guest belongs to exactly one household.
Household types:
  Named household: "The Paine Family 1"
    Groups specific people together regardless of last name.
    The trailing number (e.g. " 1") is stripped from display names.
    Display: "The Paine Family"
  Wildcard household: "* Lawson" (name starts with *)
    Groups everyone with last_name = "Lawson" together.
    Searching for any Lawson returns the whole Lawson family group.
    Display: "The Lawson Family"

The `*` wildcard households exist for families where all members of a last
name should RSVP together but were not individually specified at import time.

Household display logic (getDisplayHouseholdName function):
  1. Strip leading "*" and trim
  2. Strip trailing space + number (regex: /\s+\d+$/)
  3. If the raw name starts with "*": display as "The {LastName} Family"
  4. Otherwise: display the stripped name

IMPORTANT: The frontend NEVER shows household numbers or the asterisk.
These are purely administrative — visible only in admin panel and DB.

--- GUEST SEARCH ---
Search input: guests ilike query on first_name, last_name, nicknames
  Query: `%${searchQuery}%` (partial match, case insensitive)
After finding matching guests, fetch ALL guests with same household_id.
This ensures the whole household is shown even if only one member was searched.

--- RSVP FORM STATE ---
For each guest in the household:
  attending:       boolean | null  (null = not yet answered)
  meal_option:     string | null
  food_allergies:  string | null
  song_request:    string | null
  advice:          string | null

Initial state: `attending: g.attending ?? null`
  (Uses whatever is in DB, or null if null/undefined. NEVER defaults to true.)

--- ATTENDING BUTTONS ---
"Attending" button:    active styling when attending === true  (exact)
"Declined" button:     active styling when attending === false (exact)
Neither button active: attending === null (guest hasn't selected yet)

--- MEAL OPTIONS ---
Currently: mealOptions array is empty in wedding-data.ts
When populated, will render a meal choice dropdown/selector per guest.
"A Few More Things" section (allergies, song request, advice) only shows
when at least one guest in the household has attending === true.

--- SUBMIT VALIDATION ---
Before submitting RSVP:
  Find first guest where attending === null
  If found: show error "Please select Attending or Declined for {first_name}."
  Block submission until all guests have explicit attending value.

--- SUPABASE WRITES ---
RSVP submission updates guests table directly via anon key Supabase client.
No server-side API route for RSVP — direct client → Supabase writes.
RLS is currently DISABLED on guests/households tables.
This is a known security gap — mitigated by the data being non-sensitive.

--- HOUSEHOLD GROUPING LOGIC ---
Wildcard (*) households: When a guest is found with household.name starting
with "*", all other guests in that household are fetched by household_id.
This is standard household behavior — the `*` just signals the grouping style.

--- GUEST TABLE SCHEMA (full columns) ---
  id              UUID PRIMARY KEY
  household_id    UUID FK → households(id)
  first_name      TEXT NOT NULL
  last_name       TEXT NOT NULL
  suffix          TEXT (nullable, Jr/III etc)
  nicknames       TEXT[] (array of alternate name spellings)
  attending       BOOLEAN (nullable — null = no response)
  meal_option     TEXT (nullable)
  food_allergies  TEXT (nullable)
  song_request    TEXT (nullable)
  advice          TEXT (nullable)
  plus_one_name   TEXT (nullable — name of their plus-one)
  affiliation     TEXT (nullable — how they know couple)
  side            TEXT (nullable — "bride" | "groom" | "both")
  likelihood      TEXT (nullable — likelihood of RSVPing yes, admin-only)
  created_at      TIMESTAMPTZ
  updated_at      TIMESTAMPTZ

--- GUEST SEED ---
File: supabase/seed_guest_list.sql
Applies to: Round 2 guest list (~192 guests, 103 households)
Command: TRUNCATE guests, households CASCADE (resets all guest data)
Then: Full INSERT for all households and guests
Requires: add_guest_metadata migration must be applied first
          (adds plus_one_name, affiliation, side, likelihood columns)
Admin context: likelihood = admin estimate of RSVP probability
               side = which side of the family (bride/groom/both)
               affiliation = how they know the couple

16. GAMES SYSTEM — FULL DETAIL

--- SHARED INFRASTRUCTURE ---

Game Player Account:
  Guests enter a username and email before submitting to the leaderboard.
  Stored in localStorage key "painedle_game_player" (shared across all games).
  Also set as a cookie with same key.
  No password. No account creation on server until first score submission.
  The "account" is just localStorage-persisted username + email.

Score Submission Flow:
  1. Game completes
  2. ScoreSubmissionForm appears
  3. Guest clicks "Submit to Leaderboard"
  4. Client reads username/email from localStorage
  5. Captures browser profile metadata (timezone, language, screen size)
  6. POSTs to /api/games/submit-score:
       game, score, maxScore, attempts, solved, puzzleKey,
       email, username, metadata (browser + request data)
  7. Server upserts game_player by email, then upserts game_score
  8. Only updates if new score beats existing (higher score, or same + fewer attempts)
  9. Returns { improved: boolean }
  10. On success: dispatches leaderboard refresh window event

Leaderboard:
  Shown next to each game (2-column layout on desktop).
  Fetches top 10 scores for the specific game + puzzle key.
  Auto-refreshes when leaderboard refresh event fires.
  No authentication needed to view leaderboard.

--- PAINEDLE (Daily Word Game) ---
Type: Wordle clone with wedding-themed words
Status: LIVE (no unlock gate)
Puzzle rotation: New word every day
  Anchor: 2026-03-08 = "bride" (word at index 0 in PAINEDLE_WORDS)
  Sequential rotation through 80 words
  After 80 days: cycles back to index 0
Word length: Always exactly 5 letters
Max guesses: 6
Persistence: Game state saved to localStorage per date ("painedle-YYYY-MM-DD")
  If you revisit the same day, your progress is restored.
  If you solved it, the solved state is shown.

Word validation:
  Uses npm "word-list" package (~500k words) server-side
  All PAINEDLE_WORDS are also valid guesses regardless of dictionary
  Custom words (wedding-themed) may not be in standard dictionary

Scoring for leaderboard:
  score = MAX_GUESSES - guesses_used + 1
  Won in 1 guess: 6 points
  Won in 6 guesses: 1 point
  Lost (didn't solve): 0 points, no leaderboard submission

Puzzle key for leaderboard: the dateKey string (e.g. "2026-03-10")
  Each day is a separate leaderboard — leaderboard filters by today's key.

Midnight reset:
  PainedleGame component sets a timeout to fire at local midnight + 50ms.
  When it fires, dateKey state updates → PainedleBoard remounts → fresh game.

  If puzzle changes (new words added), update this key to reset the leaderboard.

17. SESSION & AUTHENTICATION
================================================================================

--- ADMIN SESSION ---
Implementation: Custom HMAC-SHA256 signed cookie. No third-party auth library.

Token format: "${role}.${expiresAt}.${base64urlHmac}"
  role:      e.g. "Master", "User 1"
  expiresAt: Unix timestamp (seconds), 14 days from creation
  hmac:      SHA256 HMAC of "${role}.${expiresAt}" using secret key

Cookie settings:
  name:     wedding_admin_session
  httpOnly: true   (not accessible via JS)
  secure:   true   (only sent over HTTPS — Vercel is always HTTPS)
  sameSite: "lax"  (prevents CSRF, allows normal navigation)
  maxAge:   1209600 (14 days in seconds)

Verification (every protected API call):
  1. Read cookie from request
  2. Parse token → role, expiresAt, receivedHmac
  3. Recompute HMAC using same secret
  4. Timing-safe compare (prevents timing attacks)
  5. Check expiry
  6. Return { role } if valid, null if invalid

Secret key source:
  Primary:  process.env.ADMIN_SESSION_SECRET
  Fallback: concat of all ADMIN_PASSWORD_* env vars (no separator)
  Always set ADMIN_SESSION_SECRET explicitly in production.

Role capabilities:
  "Master":  All admin tabs + Security tab + AdminEditBar edit mode
  "User 1-5": RSVP tab + Games tab + Content tab only
             No security logs, no edit mode

--- GAME PLAYER "ACCOUNTS" ---
Not real accounts in the authentication sense.
Just localStorage + cookie storage of { username, email }.
No password. No server-side session.
The email is used as the unique key when upserting game_player records.
Players can change their username by updating Account Settings.

--- SUPABASE AUTH ---
NOT USED. The project uses Supabase as a PostgreSQL database only.
No Supabase Auth features (users, sessions, magic links) are enabled.
All auth is the custom HMAC cookie system above.

20. KNOWN QUIRKS & GOTCHAS
================================================================================

--- TAILWIND V4: FONT UTILITIES CANNOT USE var() IN @theme ---
PROBLEM: Tailwind v4's @theme block processes values at computed value time.
Nesting CSS var() references inside @theme for font-family causes IACVT
(Invalid at Computed Value Time) — the font simply doesn't apply.

SYMPTOM: Fonts appear as system default (Arial/sans-serif) instead of Bodoni
Moda or Montserrat.

SOLUTION: Define font utilities in @layer utilities instead of @theme:
  @layer utilities {
    .font-heading { font-family: var(--font-playfair), serif; }
    .font-body    { font-family: var(--font-inter), sans-serif; }
  }
This is already implemented correctly. If fonts break in future, check this.

--- FONT CSS VARIABLE NAMES ARE HISTORICALLY WRONG ---
--font-playfair is actually Bodoni Moda (not Playfair Display)
--font-inter is actually Montserrat (not Inter)
These were named in early sessions and never corrected. They work fine —
just don't be confused when reading globals.css or layout.tsx.

--- TURBOPACK CACHE MUST BE CLEARED AFTER MAJOR FILE SYNCS ---
When AI makes many changes in a worktree and those are synced back to main,
Turbopack's incremental cache (.next/) can have stale state.
Symptoms: changes don't appear, type errors seem wrong, weird 404s.
Fix: rm -rf .next && npm run dev

--- WORKTREE .env.local IS SEPARATE FROM MAIN PROJECT ---
The git worktree at .claude/worktrees/sweet-robinson/ has its own .env.local.
If Supabase shows "placeholder" errors or RSVP fails to load:
  cp /path/to/main/project/.env.local /path/to/worktree/.env.local
  Restart dev server.

--- RSVP USES CLIENT-SIDE SUPABASE (NO SERVER ROUTE) ---
The RSVP page reads and writes directly to Supabase using the anon key.
This means the anon key has write access to guests and households tables.
RLS is currently disabled. This is intentional (simplicity) but is a gap.
Do NOT enable RLS on guests without adding appropriate policies first —
the RSVP form will break if RLS is enabled without a corresponding policy.

--- game_players.email IS THE UNIQUE KEY ---
If a guest changes their email in GameAccountPanel, their score history
is NOT merged — a new player record is created.
Their old scores remain under the old email.
Only relevant if a guest changes email between sessions.

--- PAINEDLE WORD ROTATION STARTS AT INDEX 0 (NOT 1) ---
PUZZLE_ROTATION_START = "2026-03-08"
Word on 2026-03-08 = PAINEDLE_WORDS[0] = "bride" (confirmed anchor)
Word on 2026-03-09 = PAINEDLE_WORDS[1], etc.
If you add words to PAINEDLE_WORDS, add them at the END to not break rotation.
If you remove or reorder words, past dates will show different words.
Never remove or reorder words from PAINEDLE_WORDS after launch.

--- TRIVIA puzzleKey IS FIXED ---
All trivia scores use puzzleKey = "wedding-day-trivia".
There is one shared trivia leaderboard for all time.
If questions change significantly, consider changing the puzzleKey to
reset the leaderboard (e.g. "wedding-day-trivia-v2").

--- CROSSWORD puzzleKey SHOULD CHANGE IF PUZZLE CHANGES ---
CROSSWORD_PUZZLE_KEY is a constant in crossword.ts.
If you change the crossword entries, increment/change this key.
Otherwise old scores (from old puzzle) will appear on new puzzle's leaderboard.

--- data-admin-key ELEMENTS MUST NOT CONFLICT ---
If two elements on the same page have the same data-admin-key value,
clicking either one will save to the same key. This could cause
unexpected behavior. Keep all keys unique within a page.
The pattern-based keys (story.item.N.*, faq.N.*, etc.) are unique by design.

--- revalidatePath REQUIRES Next.js 13.4+ APP ROUTER ---
The call revalidatePath("/", "layout") after site_settings changes works
because this is an App Router project. Do not move API routes to Pages Router.
Also: revalidatePath only affects ISR/static cache. Dynamic server components
(which this project uses) re-render on every request anyway, but Next.js
route caching still requires invalidation for proper behavior.

--- SUPABASE SERVICE ROLE KEY IS SERVER-ONLY ---
SUPABASE_SERVICE_ROLE_KEY must NEVER be exposed to the browser.
It is not prefixed with NEXT_PUBLIC_, so Next.js will not bundle it for client.
If you accidentally add NEXT_PUBLIC_ prefix, the key will be exposed.
Verify: any file importing this key must be server-only (no "use client").

--- NEXT.JS 16 + REACT 19: useEffectEvent ---
PainedleGame.tsx uses useEffectEvent from React (experimental in React 18,
stable in React 19). This hook is used for the keyboard event handler to
avoid stale closure issues. If you downgrade to React 18, this will break.
Do not change React version.

--- useSyncExternalStore FOR LOCALSTORAGE ---
GameAccountPanel uses useSyncExternalStore to safely read localStorage
on the client without causing hydration mismatches. This is the correct
pattern for any component that reads localStorage in Next.js App Router.
The server snapshot always returns null/empty. Client snapshot reads localStorage.

--- SUPABASE ANON KEY IN BROWSER: WHAT IT CAN ACCESS ---
Current RLS state:
  households:        No RLS — anon can read AND write (RSVP needs this)
  guests:            No RLS — anon can read AND write (RSVP needs this)
  admin_logs:        No RLS — but anon key is never used for this table
  game_players:      No RLS — anon can read (leaderboard) 
  game_scores:       No RLS — anon can read (leaderboard)
  trivia_questions:  RLS enabled — anon can only SELECT non-archived rows
  site_settings:     RLS enabled — anon cannot access at all

--- VERCEL BODY SIZE LIMIT ---
Vercel serverless functions have a 4.5MB request body limit.
Image uploads from AdminEditBar/ContentAdminPanel pre-compress before sending:
  Max dimension: 2400px
  JPEG quality: 0.92
This keeps uploads well under 4.5MB for typical photos.
For very large raw images, compression may need to be more aggressive.

--- CROSSWORD GATE TIMEZONE ---
CROSSWORD_UNLOCK_AT = new Date("2026-09-19T00:00:00-05:00")
TRIVIA_UNLOCK_AT    = new Date("2026-09-26T00:00:00-05:00")
Both use -05:00 (Central Standard Time).
Note: September is in Daylight Saving Time (CDT = UTC-5).
Verify: CDT is actually UTC-5 (not UTC-6 which is CST).
Central Daylight Time (CDT) = UTC-5. Central Standard Time (CST) = UTC-6.
Sep 19 and Sep 26 are both in CDT, so -05:00 is correct for midnight CDT.

--- IMAGE OVERLAY SYSTEM ---
Overlays are CSS overlay divs, not actual image composition.
An image with overlay { color: "#1A3F6F", opacity: 0.4 } shows a semi-transparent
navy div over the image (z-index above image, pointer-events none).
This is purely cosmetic and does not modify the actual image file.
Overlays are stored in site_settings as part of the image key's value:
  { main: "https://...", overlay: { color: "#1A3F6F", opacity: 0.4 } }

--- ADMIN_LOGS TABLE HAS NO RLS ---
admin_logs stores login events. It's written by the auth API route using
the service role key, so anon access doesn't matter.
No RLS needed — anon can't write to it (only service role key writes it).

--- HOUSEHOLD SEED: TRUNCATE IS DESTRUCTIVE ---
Running seed_guest_list.sql starts with:
  TRUNCATE guests, households CASCADE;
This deletes ALL RSVP responses. Only run this before the website goes live
or when you want to completely reset the guest list.
After applying seed in production, any existing RSVPs will be lost.
