This review covers visual design, layout, page-by-page UX, content readiness, mobile behavior, accessibility instincts, SEO signals, frontend quality, and backend/admin reliability. I reviewed the codebase, ran a production build, and visually inspected key pages on desktop and mobile.

Executive Summary

The strongest thing about this site is that it already has a point of view. It does not feel auto-generated. The color palette, typography, hero photography, and editorial tone all work together to create a polished, warm, upscale wedding experience. The RSVP entrance is especially memorable, the attire page is visually strong, and the games layer gives the site more personality than most wedding websites have.

Where the site loses points is mostly in the practical layer. A guest should never have to wonder where the schedule lives, whether details are final, or whether a practical page is still under construction. There are also a few implementation decisions that are fine for a private personal project but not ideal from a professional robustness standpoint: public-facing mutation endpoints, basic password-only admin auth, swallowed server errors, and a lint workflow that appears to crawl extra generated worktree output.

What is working best:

The visual identity is cohesive, the site feels premium, and the main guest journey is emotionally on-brand. The codebase also shows real structure rather than being a pile of page-specific hacks.

What most needs attention:

Make the practical information harder to miss, remove any traces of incompleteness before invites go out, and tighten the admin / data-writing surfaces.

Backend / Admin Review

For a personal event site, the backend is more capable than average. Pulling editable content from Supabase and wiring an admin surface for images, page visibility, trivia, and guest management is a legitimate strength. That said, this layer is also where the project is least “luxury” and most “startup practicality.”

  • Good: Centralized content/data model, clear app structure, editable settings, and successful production build.
  • Good: Page visibility controls and admin editing hooks show real operational thinking.
  • Concern: Admin access is password-only and intentionally lightweight. That is okay for a small private site, but not a high-security model.
  • Concern: The RSVP viewed endpoint updates guest records from public input with only a household id. That may be acceptable for low-risk telemetry, but it is still a writable public surface.
  • Concern: Several server-side helpers swallow errors and fall back quietly. Good for uptime, less good for diagnosing bad data or broken integrations.
  • Concern: The lint command did not finish cleanly in a reasonable window because it appeared to be traversing generated `.next` output inside a `.claude/worktrees/...` folder. That is a tooling hygiene issue worth tightening.

Biggest Things Helping the Grade

  • The site has a real visual identity and does not look templated.
  • The RSVP and attire experiences feel custom and emotionally on-brand.
  • Travel and explore content provide genuine guest value, not filler.
  • The admin/content architecture is practical and gives the site a real maintenance path.
  • The games concept gives the website personality and makes it memorable.

Biggest Things Holding the Grade Back

  • Some important logistical pages are not surfaced as aggressively as they should be.
  • Any remaining “coming soon” or placeholder-style content reduces guest trust immediately.
  • The practical pages are slightly less visually resolved than the romance-driven pages.
  • Mobile spacing is good overall but not yet effortless everywhere.
  • The backend/admin layer is useful, but security and tooling are still more functional than refined.

Priority Roadmap

Quick Changes

Low effort, high return

  • Add stronger site-wide paths to Wedding Details, Schedule, and FAQ.
  • Remove or finish every “coming soon” fragment before guest traffic ramps up.
  • Raise contrast on lighter secondary text and subtle nav states.
  • Make the RSVP mobile card breathe more with slightly tighter headline scale and more form spacing.
  • Give the travel map block a more definitive, polished fallback if embeds fail or look blank.
Medium Changes

Noticeable quality jump

  • Rework the homepage lower section so logistics are surfaced more assertively without losing the elegance.
  • Turn FAQ into accordions or grouped topics so it feels designed, not just stacked.
  • Create a stronger “guest essentials” block that combines venue, ceremony time, schedule, and RSVP deadline in one obvious spot.
  • Scope the lint configuration so generated artifacts and side worktrees are excluded.
  • Add more explicit empty/error states around admin-driven content so quiet failures are easier to catch.
Hefty Changes

Would move the site toward A / A+

  • Tighten admin authentication with stronger session handling, rate limiting, and better audit protections.
  • Harden public write endpoints so guest-related mutations are more constrained and observable.
  • Build a unified practical-information design system for details, schedule, travel, FAQ, and RSVP support states so the utility pages feel as premium as the showcase pages.
  • Add a true launch-readiness checklist to detect incomplete content, hidden pages, missing travel data, and stale settings before publishing.

Final Verdict

Final grade: B+. This is already better designed than most custom wedding websites and dramatically better than most template-driven ones. The brand, imagery, and overall tone are strong enough that the site feels memorable.

The path to an A is clear: make the utility layer as polished and as unmistakable as the emotional layer, remove every trace of unfinished content, and tighten the backend edges so the site feels as reliable behind the curtain as it looks on the front.
